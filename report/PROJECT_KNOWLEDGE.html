Part I

The big picture — what and why

Imagine a bank’s customer-support inbox. Thousands of short messages arrive: “my card hasn’t arrived,” “why was I charged twice,” “how do I change my PIN.” Each needs to be routed to the right workflow. There are 77 distinct categories (called intents) in the dataset we use. A model that reads the message and picks the right intent could automate a huge amount of this work.

But automation has a catch: a model that is wrong and doesn’t know it is worse than no automation, because a confidently wrong answer sends the customer down the wrong path. The valuable question is not “can the model be right most of the time?” It is “when the model is about to be wrong, can it tell, and hand the ticket to a human instead?”

In plain English · Selective prediction

Selective prediction (also called “prediction with a reject option”) means the model is allowed to say “I don’t know” and abstain instead of guessing. Think of a student in an exam that penalizes wrong answers: a smart student leaves blank the questions they’re unsure of, so their answered questions have a much higher score. We want the model to do the same — skip the hard tickets so the ones it answers are reliably correct.

Two numbers describe such a system, and they trade off against each other:

Coverage
the fraction of tickets the model chooses to answer (its automation rate)
Selective risk
the error rate among the answered tickets only

Answer everything and you get high coverage but you’re stuck with the model’s raw error rate. Answer only the tickets the model is most sure about and the error rate among them plummets — but you automate fewer tickets. The art is choosing where on that curve to sit, and being able to promise how accurate the answered set will be.

Story · Why this project exists

Most “LLM portfolio” projects are some flavour of “I built a chatbot.” They demonstrate that you can call a model. They rarely demonstrate statistical judgement. This project was deliberately designed to be different: the classifier is only the setup; the real subject is the layer on top that turns a fallible model into a controllable one with a defensible guarantee. That combination — a generative model, real serving, and honest uncertainty quantification — is the thing that makes it stand out.

The whole system runs on a laptop — an RTX 4060 with 8 GB of video memory — not a cloud cluster. That constraint shaped every technical choice, and being able to explain those constraints is itself part of the story.

Part II

The concepts, explained from scratch

Before the phase-by-phase story, here is every idea the project rests on, built up from nothing. If you already know a concept, skim it; if an interviewer asks “explain this like I’m not technical,” these are your answers.

Language models, “instruct” tuning, and tokens

In plain English · LLM & tokens

A large language model (LLM) is a program that, given some text, predicts what text is likely to come next. It works in tokens — chunks of text roughly the size of a short word or word-piece (“card” is one token; “withdrawal” might be two). An instruct-tuned model has been further trained to follow instructions and hold a conversation, rather than just autocomplete. We use Qwen2.5-3B-Instruct: “3B” means about 3 billion internal parameters — small as modern LLMs go, which is exactly why it fits on a laptop.

Under the hood · Why this exact model

Qwen2.5-3B-Instruct wasn’t picked at random. It is natively implemented inside the transformers library — its model code ships with the library, so it can’t drift out of sync. The first choice, Phi-4-mini-instruct, relied on trust_remote_code (model code downloaded from the hub and executed), and that bundled code broke twice across library versions — first an import error, then a generation-shape error. Switching to a natively-supported model removed a whole class of fragility. And a slightly weaker 3B model is a feature, not a bug here: it produces a richer accept/abstain trade-off than a near-perfect classifier would. The uncertainty layer on top is model-agnostic — swap in a stronger model and the same machinery applies unchanged.

4-bit quantization — making a big model fit in a small memory

In plain English · Quantization

Each of the model’s billions of parameters is a number. Normally each is stored in 16 or 32 bits of memory. Quantization stores them in fewer bits — here just 4 bits each (a scheme called NF4) — which shrinks the model roughly 4× so it fits in 8 GB of GPU memory. It’s like saving a photo as a smaller JPEG: a little quality is lost, but it fits and still looks fine. The whole model must sit in the GPU’s memory (VRAM); if it spills over to normal RAM, everything slows down by ~30×.

Scoring labels vs. generating text

There are two ways to get a classification out of an LLM, and the choice matters.

  • Generation — ask the model to write the answer (“the intent is: card arrival”). Simple, but you only see the words, not how confident the model was.
  • Label scoring — for each of the 77 candidate intents, ask the model “how likely is this exact label to follow the prompt?” and read off the probability. This is the path we use, because it hands us a confidence number for every class, which is the raw material for everything later.
In plain English · Log-probability & softmax

The model assigns every possible next token a probability. A log-probability (“logprob”) is just the logarithm of that probability — used because multiplying many small probabilities underflows, whereas adding their logs is stable. To score a whole label like “lost or stolen card,” we add up the logprobs of its tokens. Softmax is the function that turns a set of raw scores into percentages that sum to 100%. So: score all 77 labels → softmax → a probability for each intent → pick the biggest.

Under the hood · Length normalization

Longer labels have more tokens, and every extra token multiplies in another probability (adds another negative logprob), which mechanically makes long labels look “less likely.” To keep the 77 labels comparable, we divide each label’s total logprob by its number of tokens — an average logprob per token rather than a sum. This is called length normalization. Without it, the model would be biased toward short intent names for a silly, non-semantic reason.

Under the hood · The KV-cache trick

Scoring 77 labels naively means running the model 77 times over the same long prompt — wasteful. When a transformer reads a prompt it builds an internal summary called the KV cache (key/value cache). We compute that once for the shared prompt, then replay just the short label tokens against it — about 10× faster. We verified this fast path produces identical numbers to the naive method (matching to ~0.000001 in high precision), so the speed-up costs no correctness.

Few-shot prompting and retrieval

In plain English · Zero-shot, few-shot, retrieval

Zero-shot: you just ask the model to classify, giving no examples. Few-shot: you paste a handful of already-labelled examples into the prompt first, so the model can pattern-match. Retrieval-augmented few-shot goes further: for each incoming ticket, you fetch the handful of past examples most similar to that specific ticket and show those. It’s the difference between studying a fixed cheat-sheet versus being handed the exact worked examples that resemble the question in front of you.

In plain English · TF-IDF & k-NN

TF-IDF is a classic way to turn text into numbers by counting words, weighting down words that appear everywhere (“the,” “my”) and up the distinctive ones (“PIN,” “exchange”). Two messages with similar distinctive words get similar number-vectors. k-NN (k-nearest-neighbours) means “find the k closest examples.” We use TF-IDF to find each ticket’s 16 nearest past examples — fast, needs no extra AI model, and works well on short texts.

Measuring how good a confidence signal is: AUROC

In plain English · AUROC

Suppose you have a confidence number for each prediction. A good confidence signal gives higher numbers to the predictions that turn out correct and lower numbers to the wrong ones. AUROC (area under the ROC curve) measures exactly that with a single number from 0.5 to 1.0. Concretely: pick one correct prediction and one wrong prediction at random; AUROC is the probability the confidence score ranks the correct one higher. 0.5 = the signal is useless (a coin flip); 1.0 = perfect separation. A signal at 0.9 is genuinely informative.

Ranking vs. calibration (a subtle, important difference)

In plain English · Calibration

A confidence signal can be a great ranker (high AUROC) yet be a poor probability. Calibration asks a stricter question: when the model says “70% confident,” is it actually right 70% of the time? A well-calibrated weather forecaster who says “70% chance of rain” is right on ~70% of such days. As we’ll see, our model’s raw confidence is a strong ranker but is under-confident as a probability — which is precisely why we don’t trust the raw number and instead calibrate a threshold from data.

The centrepiece: conformal prediction & a real guarantee

This is the intellectual core, so it gets the most careful explanation.

In plain English · The guarantee we want

We want to pick a confidence cut-off (a threshold) so that we can honestly say: “Among the tickets we answer, at least 95% are correct.” The trick is that we must choose this threshold on past data but promise it holds on future data we haven’t seen. Getting that promise right — with maths that works for a finite amount of data, not just “in the limit” — is what conformal calibration provides.

In plain English · Exchangeability

The one assumption behind the guarantee is exchangeability: the past (calibration) data and the future (test) data come from the same distribution — roughly, tomorrow’s tickets look statistically like today’s. It’s a weaker cousin of “independent and identically distributed.” If the world drifts (a new product launches, slang changes), the promise weakens and you must re-calibrate. We state this assumption openly rather than pretend it away.

Under the hood · Learn-then-Test, in four steps

The method (called Learn-then-Test, part of the RCPS family) is a hypothesis test done once per candidate threshold:

  1. Propose a threshold. For a given cut-off, look at the calibration tickets it would accept: say n of them, of which k are wrong.
  2. Test it. Ask: “If the true error rate here were as bad as our target allows (say 5%), how surprising is it to see only k errors out of n?” That surprise is an exact p-value from the binomial distribution — no approximation.
  3. Correct for many tries. We test many thresholds, so we tighten the bar (a Bonferroni correction) so the whole search stays trustworthy at once.
  4. Choose. Among thresholds we can certify, take the one that accepts the most tickets (highest coverage).
In plain English · p-value & Bonferroni

A p-value answers “if my pessimistic assumption were true, how likely is data this good by luck?” A tiny p-value means the good result is hard to explain by luck, so we trust it. Bonferroni correction handles the fact that if you try many things, one will look good by chance; you compensate by demanding stronger evidence from each try (divide the allowed luck-budget by the number of tries). It is the statistical version of “if you buy 100 lottery tickets, winning once isn’t impressive.”

In plain English · The risk-coverage curve

Sweep the threshold from “answer everything” to “answer only the surest,” and plot accuracy-of-answered against fraction-answered. That’s the risk-coverage curve. It shows the whole menu of trade-offs at once; you then pick one operating point on it and justify the choice, instead of cherry-picking a single flattering number.

Part III · The journey

Phase 1 Building a classifier that works

The plan: load the banking77 dataset, split it cleanly, build the label-scoring machinery, and measure raw accuracy. The target was a “sane” 70–85% for a model this size — the anchor number the whole project hangs from.

Under the hood · Clean data splits

The 13,083 messages were divided into three disjoint pools: 8,003 as a “few-shot pool” (the only place example prompts may be drawn from), 2,000 for calibration (choosing the threshold), and the canonical 3,080 for test (final scoring only). Keeping these separate is a non-negotiable rule: if example prompts or the threshold ever “saw” the test data, the results would be inflated and meaningless.

Story · A disappointing 44%, and the detective work

The first honest measurement was a shock: 44.4% — far below the 70–85% expected. The temptation is to start randomly tweaking. Instead, the scoring maths was first proven correct (the fast path matched a slow reference method to six decimal places), which ruled out a bug. Then a controlled bake-off compared prompt styles and scoring variants head-to-head on the same examples. The verdict was decisive and the fix was not a hack — it was retrieval.

ConfigurationTest accuracy
Zero-shot (no examples)44.4%
77 fixed examples (one per intent)64.7%
16 retrieved examples (chosen)89.4%

Retrieval wins because banking77 is full of near-twins — card_arrival vs. card_delivery_estimate, for instance. Showing the model the neighbours of the specific ticket demonstrates exactly the distinction it needs to make. It is also faster than a giant fixed prompt, because the prompt stays short.

Result · The honest headline

89.4% raw top-1 accuracy — above the target band. But honesty demanded one more check.

Story · The baseline that keeps you honest

A 90% number could be a fake headline. What if the retrieval alone — with no LLM — already got you most of the way? So a plain TF-IDF vote over the same retrieved neighbours was run: it scored 80.6%. That means the LLM’s genuine contribution is +8.8 points — and a paired statistical test (McNemar, p = 0.00000057) confirms the difference is real, not luck. The honest framing: retrieval supplies the candidates; the LLM’s job is discriminating among the confusable ones. Both numbers belong in the write-up. An interviewer will respect that far more than a bare 89%.

In plain English · McNemar’s test

When two systems are graded on the same items, McNemar’s test looks only at the items where they disagree — where the LLM was right and the baseline wrong, versus the reverse. If those disagreements lopsidedly favour the LLM, the improvement is real. Here 61 flips favoured the LLM and only 17 the baseline — a lopsidedness that’s essentially impossible by chance.

Two “negative results” were also kept, because they show rigour: a popular trick called PMI/prior-correction was tried and hurt accuracy (so it was dropped), and free-form generation (46.7%) scored worse than label scoring — independently confirming the scoring path wasn’t hiding a bug.

Phase 2 Three ways to measure confidence

The plan: the classifier now makes a prediction; we need a confidence for each prediction so we can later decide which to trust. Three different signals were computed for the same prediction, so the comparison is apples-to-apples.

In plain English · The three signals

1. Logprob confidence — straight from the scoring step. Two flavours: the winning label’s probability (“max-softmax”), and the margin — how far ahead the winner is over the runner-up. A big margin means “clear winner.”
2. Self-consistency — ask the model the same question 5 times with a dash of randomness (“temperature”); if it says the same thing every time, it’s confident; if it wavers, it isn’t.
3. Verbalized confidence — simply ask the model to state a number: “how confident are you, 0 to 1?” The cheapest and, spoiler, the worst.

In plain English · Temperature

Temperature controls randomness when a model generates text. At temperature 0 it always picks its top choice (deterministic). Higher temperatures let it sometimes pick lower-ranked words, adding variety. Self-consistency needs some randomness (we used 0.7) so the 5 answers can differ when the model is genuinely torn.

Story · Is self-consistency even working?

An early check found the 5 samples were identical for most tickets — suspicious. Was randomness broken? A quick probe cranked the temperature to an extreme (5.0) and the model duly produced gibberish, proving the machinery worked. The real explanation was simply that this model is very decisive: on easy tickets all 5 samples agree, and only on genuinely ambiguous ones do they diverge. That’s honest behaviour — and it foreshadows why self-consistency turns out to be a coarse signal.

Result · A clean ranking that held on both splits

Measured by AUROC (correct-vs-wrong separation) on the test split:

Confidence signalAUROCVerdict
Logprob margin0.915strong — the winner
Logprob max-softmax0.909strong
Self-consistency0.650modest
Verbalized0.518~coin flip — useless

The lesson is quotable: the model’s log-probabilities know when it’s wrong; the model’s words do not. Asking a model to rate its own confidence produces a number that’s barely better than a coin flip — it says “0.95” whether right or wrong. This is a genuine, well-known failure mode, and demonstrating it cleanly is a legitimate finding, not a disappointment.

Phase 3 The guarantee — the heart of the project

The plan: turn the best confidence signal into a threshold with a real, finite-sample guarantee, and report the whole trade-off curve. This phase needs no GPU — it’s pure statistics on the cached numbers from Phase 2.

Story · A bug the tests caught before it shipped

The first implementation walked candidate thresholds from the “safest” end (accept only the single most confident ticket) downward. It failed instantly — and a unit test caught it. The reason is a genuine statistical insight: with only one accepted ticket you have zero power to prove a 95% rate, so the sequential test stalls at the very first step. The fix was to switch to Learn-then-Test with a Bonferroni correction over a grid of thresholds, which sidesteps the ordering problem entirely. Being able to tell this story — a subtle bug, caught by a test, fixed with the right method — is gold in an interview.

Result · The operating point

Using the logprob-margin signal with a target of ≥95% accuracy on answered tickets, the threshold calibrated on calibration data and frozen, then applied to the untouched test set:

Read that as a sentence: answer four out of five tickets at 98% accuracy, and route the hard one-in-five to a human — with a calibrated promise behind it. The guarantee was also checked at other targets; every certified case held its promise on the test set.

Under the hood · Why it’s a bit conservative

The target was 95% but the achieved accuracy was 98.1% — the method over-delivers. That’s the price of a distribution-free, finite-sample guarantee: it leaves a safety margin, so the reported coverage is a floor, not a ceiling. An interviewer may probe this; the honest answer is “yes, it’s conservative by design, and with more calibration data the margin shrinks.”

Under the hood · Why the naive shortcut is unsafe

The tempting shortcut is: “just pick the threshold whose error on calibration data is already 5%.” On a single test draw it often looks fine — but it sits right at the edge with no margin. A 300-run simulation (in the test suite) shows this naive rule breaches its target far more often than allowed, while the calibrated method stays within bounds. That contrast — a proper procedure vs. a plausible-looking shortcut — is the whole reason conformal methods exist.

Calibration ≠ probability: the reliability check

Why not just fix the probabilities directly?

A natural objection: the reliability diagram shows the confidence is a bad probability, so why not repair the probability and threshold that? There is a whole family of post-hoc calibration methods that do exactly this.

In plain English · Temperature / Platt / isotonic scaling

These are recipes that take a model’s miscalibrated confidence numbers and re-shape them so they mean what they say. Temperature scaling divides the scores by one learned number to soften or sharpen them. Platt scaling fits a little logistic curve mapping raw score → true probability. Isotonic regression fits a flexible staircase that only has to go uphill. All three learn their fix on held-out data, and all three would genuinely make our 0.35 mean “~0.35 correct.”

Under the hood · Why they’re not enough here

They give you a better-looking number, not a guarantee. If you temperature-scale the softmax and then accept everything above 0.95, you still have no finite-sample promise that the accepted set is 95% accurate on future tickets — you’ve only reshaped the score, and the reshaping itself is estimated from finite data with its own error. Conformal / Learn-then-Test is orthogonal and stronger: it makes a distribution-free, finite-sample promise about the error rate of the accepted set, and it needs only the score’s ranking, not its numeric calibration. The two even compose: you could Platt-scale first for interpretable numbers, then conformalize for the guarantee. But the guarantee is the thing only conformal provides — which is why it, not scaling, is the heart of the project.

Phase 4 Serving it as a real service

The plan: wrap the frozen system in an API, a batch evaluator, figures, and a readable write-up — so a stranger understands it in five minutes.

The service is a small FastAPI web endpoint. It loads the model once at startup, then for each ticket it retrieves neighbours, scores the 77 labels, computes the margin, and returns a decision. It was smoke-tested live — these are real responses:

Production realities — latency, throughput, and operation

A portfolio project should be honest about what it would take to run for real.

Under the hood · Where the time goes

Each ticket costs roughly 1.5–2 seconds on the laptop GPU. That time is dominated by scoring the 77 candidate labels — even with the KV-cache trick, that’s the bottleneck. Retrieval is milliseconds (TF-IDF is cheap), and the model loads once at startup (~2 minutes) and is reused. So this is a good fit for asynchronous triage — routing a ticket within a couple of seconds — not for sub-100 ms real-time autocomplete.

In plain English · How you’d make it faster

Four honest levers: batch several tickets through the GPU together to amortize cost; prune the candidate labels — use the retriever to score only the ~10 plausible intents instead of all 77; use a bigger GPU (this was deliberately built on an 8 GB laptop); or distil the classifier into something lighter. The uncertainty layer is unaffected by any of these — it operates on whatever confidence scores come out.

In plain English · Operating it over time

You freeze the threshold and then watch for drift. Concretely: monitor the distribution of confidence scores and the realized coverage on live traffic; if they move, that’s a signal the world has shifted and the exchangeability assumption is weakening. When it does, re-calibrate on a fresh batch of labelled tickets. The guarantee is only as current as the calibration data behind it — and saying so plainly is part of the professionalism.

Phase 5 Where the guarantee hides failures

The plan (a stretch goal): the Phase 3 guarantee is marginal — it controls the error rate pooled over all 77 intents. Averages can hide trouble. Does some subgroup of intents quietly do worse?

In plain English · Marginal vs. group-conditional

A marginal guarantee is like a school reporting a 95% overall pass rate. That average can hide one class that’s failing badly. A group-conditional guarantee instead promises the bar is met within each group. We split the 77 intents into 8 functional groups (cards, transfers, payments, top-up, cash, FX, identity, account) and checked each one.

Result · Found it, fixed it, and named the limit

The marginal threshold left the transfers group at 85.6% — a real hidden failure. Per-group calibration restored it to 92.7% and certified 7 of 8 groups. And the honest limitation, reported not hidden: at the stricter 95% target, 0 of 8 groups could be certified — because each group only has ~150 calibration examples, and certifying a high bar needs more data. Subgroup guarantees cost calibration data proportional to the number of groups. That sentence is a senior-level insight.

Part V

Interview grilling — the hard questions

Rehearse these out loud. Each answer is written the way you’d want to say it: direct, honest, and showing you understand the why.

On the statistics

Interview grilling

What exactly does your guarantee promise, and what could break it?

With probability at least 90% over the calibration draw, the true error rate among answered tickets is at most my target. The single assumption is exchangeability — test tickets are drawn like calibration tickets. It breaks under distribution shift: a new product, seasonal slang, a different customer base. The mitigation is periodic re-calibration on fresh labelled data. I never claim it’s assumption-free.

Why conformal / Learn-then-Test instead of just thresholding the softmax?

Because the softmax isn’t a calibrated probability — my reliability diagram shows it’s under-confident. Thresholding it directly gives no guarantee. Learn-then-Test is distribution-free and finite-sample: it makes a valid promise about error among the accepted set without assuming the score means anything numerically, only that higher is more confident.

Your target was 95% but you got 98%. Isn’t that a waste of coverage?

It’s the built-in conservatism of a finite-sample guarantee — it keeps a safety margin so the promise holds with high probability. The reported coverage is a floor. More calibration data shrinks the margin and buys back coverage; that’s a concrete next step, not a flaw.

Why Bonferroni? Isn’t it overly conservative?

Yes, mildly — it’s the price of testing many thresholds at once with a simple, obviously-valid correction. I chose it over fixed sequential testing because the low-risk end of my threshold range has no statistical power (few accepted points), which makes sequential testing stall. Bonferroni sidesteps that and is easy to verify. A tighter valid procedure exists and would recover some coverage.

Why not temperature-scale or Platt-calibrate the softmax and threshold that?

Because post-hoc scaling gives me a better-looking number, not a guarantee. Temperature/Platt/isotonic reshape the scores so 0.9 means 90% — useful, and they’d fix my under-confidence — but thresholding a calibrated probability at 0.95 still carries no finite-sample promise that the accepted set is 95% accurate on future data. Conformal is orthogonal and stronger: it controls the accepted-set error rate directly, distribution-free, using only the score’s ranking. You can even compose them — scale for interpretable numbers, then conformalize for the promise.

On the model & data

Interview grilling

Your no-LLM baseline gets 80.6%. So is the LLM even needed?

Retrieval alone gets you to 81%; the LLM adds a statistically significant +8.8 points (McNemar p < 1e-6) by discriminating among near-synonymous intents that a bag-of-words vote confuses. I report both numbers precisely so the LLM’s value is neither overstated nor hidden. The framing is “retrieval proposes, the LLM decides.”

Why a tiny 3B model? Wouldn’t GPT-4 crush this?

Two reasons. Practically, the whole thing runs on an 8 GB laptop — local, private, free to run, no API. Intellectually, a slightly weaker model produces a richer accept/abstain trade-off; a near-perfect model would make selective prediction boring. The project’s contribution is the uncertainty layer, which is model-agnostic — swap in a bigger model and the same machinery applies.

How do you know your fast label-scoring is correct?

I proved it. The KV-cache path was checked against a naive full-recompute reference in high precision — they matched to about 1e-6, i.e. floating-point noise. So the ~10× speed-up is exact, not an approximation. I also confirmed free generation scored worse than scoring, which rules out a silent scoring bug.

Isn’t 16 retrieved examples just leaking labels?

No — the retrieval index is built only from the few-shot pool, which is disjoint from both calibration and test. A test ticket can never retrieve itself or anything from the evaluation sets. Split hygiene was a design rule from day one.

Why Qwen2.5-3B specifically, and not some other small model?

Because it’s natively implemented in the transformers library — its model code ships with the library and can’t desync from it. My first pick, Phi-4-mini, relied on downloaded-and-executed trust_remote_code, and that code broke twice across library versions. Removing that dependency removed a whole class of reproducibility risk. It also pins cleanly to a commit hash. The choice was about engineering robustness, not leaderboard rank.

On honesty & judgement

Interview grilling

What’s the weakest part of this project?

Sample size. I calibrated and tested on 1,000 examples each; the per-group analysis has only ~150 per group, which is why 95% group-conditional certification wasn’t achievable. Everything is honest about that — I report where the data runs out rather than papering over it. The fix is simply more labelled calibration data.

You found verbalized confidence is useless. Did you consider you prompted it badly?

Fair challenge. It’s a known, robust finding in the literature that small models are poorly calibrated when asked to self-report, so I expected it — but I measured it rather than assumed it, on the same tickets as the other signals, and it landed at AUROC 0.52. Better prompting might lift it a little; it won’t rival the log-probs, which come from the model’s actual internal distribution.

Walk me through a decision where you changed course.

The 44% start. Rather than tweak randomly, I first proved the scoring maths correct, then ran a controlled bake-off of prompt and normalization variants on shared examples. That pointed cleanly at retrieval, which took me to 89%. The discipline — diagnose before you fix — is the point.

On engineering

Interview grilling

How is this reproducible?

Fixed seeds for every split and sample; the model revision is pinned to a commit hash; every model output is cached by a key of (example, signal, parameters) so re-runs never re-query the GPU blindly and never drift. The calibration is deterministic. Any number in the report can be regenerated from the repo.

Why cache raw logprob sums instead of the final normalized score?

So that trying a different normalization (mean vs. sum vs. prior-corrected) costs zero GPU time — it’s a pure re-computation from cached raw numbers. That decision is what made the Phase 1 bake-off cheap and fast.

On production

Interview grilling

What’s the latency, and is this real-time?

About 1.5–2 seconds per ticket on the laptop GPU, dominated by scoring the 77 candidate labels — retrieval is milliseconds and the model loads once at startup. So it’s built for asynchronous triage, routing within a couple of seconds, not sub-100 ms real-time. To speed it up I’d batch tickets, prune candidates via the retriever so I score ~10 labels not 77, move off the 8 GB laptop, or distil the classifier. The uncertainty layer is unaffected by any of that.

How would you run and maintain this in production over months?

Freeze the threshold, then monitor for drift: track the live distribution of confidence scores and the realized coverage. If they move, the exchangeability assumption is weakening — a new product, changed customer mix — and I re-calibrate on a fresh batch of labelled tickets. The guarantee is only as current as its calibration data, so scheduled re-calibration plus drift alerts is the operational contract.

What happens to the abstained 20% — is that just cost?

It’s cost avoided, not cost created: those tickets would have gone to a human anyway in a no-automation world. The lever is business-tunable — tighten the target and you automate less but with higher accuracy; loosen it and you automate more. The risk-coverage curve is literally the menu the business chooses its operating point from.
